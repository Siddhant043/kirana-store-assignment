# SQLAlchemy (async) + Alembic for persistence

The persistence layer uses async SQLAlchemy (ORM models over asyncpg) with Alembic for numbered, reviewable migrations, rather than raw asyncpg + hand-written SQL. This gives typed domain models, autogenerated-then-reviewed migrations, and a consistent session/transaction API, at the cost of some ORM ceremony over the exact locking SQL. The concurrency guarantees from ADR-0005 are preserved by expressing row locks explicitly as `select(...).with_for_update()` and keeping the deterministic `product_id`-ordered locking discipline — the ORM is not allowed to obscure that the oversell/idempotency checks happen under a held row lock. Considered raw asyncpg + SQL migrations (more transparent locking) and rejected it for the schema-management and typing ergonomics of SQLAlchemy across ~10 tables.
