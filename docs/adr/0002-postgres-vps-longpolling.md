# Postgres via docker-compose, deployed on a VPS with Telegram long-polling

The store runs as a single async bot process plus a Postgres container defined in one `docker-compose.yml` — the *same* artifact reviewers run locally and we run on a VPS. Postgres (over SQLite) is chosen for real row-level locking (`SELECT ... FOR UPDATE`) to back the concurrency guarantee. Telegram uses long-polling rather than webhooks: an always-on VPS process needs no public URL or TLS, and the compose file stays identical between local and prod. (We initially considered Render + webhook, but a VPS + long-polling gives one coherent, portable deployment.)

**VPS size: 2 vCPU / 4 GB RAM / 100 GB SSD.** The deciding factor is cores, not RAM: invoice PDF (WeasyPrint) and analytics deck (python-pptx) generation is synchronous CPU work, and on a single core a render stalls the agent loop. The second core lets a heavy render run while the loop stays responsive; 4 GB gives headroom for Postgres + agent + concurrent renders (and room to lazy-load a local Whisper fallback). A 1 vCPU / 2 GB box runs only if doc-gen is strictly serialized and voice/vision stay offloaded to APIs; 4 vCPU tiers are unused overkill for a single-owner bot.
