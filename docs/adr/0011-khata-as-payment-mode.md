# Credit bills: khata is a payment mode, charged atomically at finalize

A bill can be paid "on credit" — `khata` is a valid payment mode alongside cash/UPI/card. When `finalize_bill` runs with `payment_mode=khata` and a resolved `customer_id`, a single transaction decrements stock *and* writes a `khata_entries` charge for the bill total, updating the customer balance — the one place the billing and khata skills transact together. The customer must be grounded via `find_or_create_customer` first (same structural rule as products), so no phantom customer can be charged. Lock ordering is fixed globally — product rows (sorted by `product_id`) then the customer row — so a concurrent khata payment plus a credit bill cannot deadlock. Idempotency carries over for free: finalize is idempotent by the draft's `open → finalized` transition, so a retried credit-bill finalize returns the existing bill without re-charging the khata. Rejected the two-step alternative (finalize unpaid, then a separate `add_to_khata`) because it leaves a window where the bill exists without its charge.
