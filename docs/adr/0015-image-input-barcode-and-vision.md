# Image inputs: barcode decode tool + vision grounded through find_product

Photos are an input adapter into the existing agent loop, in two paths. Barcode photos are decoded by a `scan_barcode(image)` tool (`pyzbar` over the `zbar` system lib) that returns the exact product matched on a new `products.barcode` (EAN/UPC) field, seeded for packaged SKUs — a barcode is an exact key, so this path proceeds silently. Product photos with no barcode use Claude's vision: the image enters the message content, the model proposes a name/brand, and it **must** call `find_product` (pg_trgm + aliases) to ground the guess — extending ADR-0008's structural-grounding rule to images, so the shop never bills from a raw visual guess. Because vision is inherently less certain than typed text or a scanned barcode, the photo-ID path **always confirms with the owner** ("Looks like Amul Butter 100g — add it?") before adding, whereas barcode matches don't. Resource fit on the 1 vCPU / 2 GB box is fine: pyzbar/zbar is negligible and vision runs on the API with no local model.
