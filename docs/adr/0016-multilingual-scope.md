# Multilingual conversation; English-scaffold invoices with Unicode content

The owner may converse (typed or voice) in Hindi, Tamil, or Hinglish; a system-prompt instruction has the model mirror the owner's language. Grounding across languages is handled in data, not code: the `aliases` table seeds common native/transliterated product terms ("chini"→sugar, "namak"→Tata Salt, "atta", "chawal"→rice) so `find_product` resolves them. For artifacts, the invoice PDF stays **structurally English** — GST invoices are conventionally English legal documents (HSN, CGST/SGST, tax breakup), so regulated terminology is not translated — but all free-text content (shop name, customer name, item names) renders correctly in native script. This requires Noto Sans + Noto Sans Devanagari + Noto Sans Tamil baked into the Docker image; WeasyPrint embeds them so the PDF is self-contained. python-pptx references fonts by name, so native script in decks is best-effort (kept label-light). Rejected fully translated Hindi/Tamil invoices for v1: translating GST labels adds layout/pluralization work and legal-term ambiguity without a real compliance benefit.
