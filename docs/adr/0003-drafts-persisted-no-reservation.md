# DB-persisted draft bills with no stock reservation; authoritative check at finalize

Draft bills are first-class DB rows (`draft_bills` + `draft_lines`), not in-memory conversation state, so they survive restart and make finalize idempotent. Stock is **not** reserved while a draft is open — `add_line` does only a soft availability check; the authoritative oversell guard runs inside `finalize_bill`'s locked transaction. Rejected reservation-at-add-line because an open-ended chat would strand locks and need expiry handling; the accepted consequence is that stock can change between add and finalize, in which case finalize legitimately refuses and the model explains why. One active draft is tracked per chat, but drafts have real ids so distinct chats/customers are independent (supporting concurrency).
