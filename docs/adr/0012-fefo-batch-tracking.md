# Stock is tracked as expiry-dated batches, consumed First-Expiry-First-Out

A product's stock is not a single quantity but a set of `stock_batches` rows — each `(product_id, batch_qty, cost_price, expiry_date, received_at)`. `receive_stock` creates a batch, capturing cost price and (optional) expiry per intake, so margins stay accurate across restocks at different costs. `products.quantity` is kept as a denormalized `SUM(batch_qty)` fast-read column, reconciled on every mutation. `finalize_bill` decrements FEFO — consuming from batches ordered by `expiry_date ASC`, spanning multiple batches as needed, writing one `stock_ledger` row per batch touched. The oversell guard becomes "requested ≤ SUM(non-expired batch_qty)"; expired batches are excluded from sellable stock and a sale that would draw on them is refused with an explanation. A `list_expiring_soon` tool surfaces near-expiry batches for reorder/analytics. This extends ADR-0005: the deterministic lock order is now product_id, then batch_id ascending, preserving the no-deadlock property. Loose/non-perishable items use a null-expiry batch so the model is never forced to invent a date. Considered quantity-plus-expiry-only batches (cost stays on the product) and rejected it because margins blur when restocking at a new cost.
