# Conversation is ephemeral; the store and preferences are durable

Per-chat conversation history is held in-memory (a session map keyed by Telegram `chat_id`) and is deliberately *not* persisted — it is ephemeral working memory. All durable state (products, drafts, bills, khata, preferences, shop profile) lives only in Postgres. Consequences: a process restart loses in-flight conversation but never loses stock/drafts/khata, and an open draft is recoverable because the model can call `view_draft` on the surviving row. `/new` clears that chat's session and starts a fresh agent conversation, with preferences and shop profile re-injected from Postgres on the first turn — this is what makes "start a /new chat and it still remembers UPI/atta" work. Draft recovery after a cold restart is passive (surfaced when the owner's next message prompts a `view_draft`), not a proactive boot-time broadcast.
